## Context

既存の `shape-clipper.tsx` はブラウザ上でSVGクリッピングが動作することの検証コードとして機能した。本設計はその知見を引き継ぎつつ、地理的コレクション体験として一から設計し直す。

主要な技術的課題は2つ:
1. **地理データの扱い**: 日本の行政区画を正確にSVG化し、マップ表示とクリッピングの両方で使えるようにする
2. **座標系の二重性**: マップ全体表示用の座標と、1エリア拡大編集用の座標は別スケールで扱う必要がある

## Goals / Non-Goals

**Goals:**
- Web アプリ (Next.js) として提供し、インストール不要で即利用可能にする
- 都道府県・市区町村レベルの行政区画を地理的に忠実なSVGパスで提供する
- マップキャンバスと個別クリップ編集の両体験を単一のUIフローに統合する
- ブラウザのみで完結する（外部サーバーへの写真送信なし）

**Non-Goals:**
- ネイティブアプリ（将来フェーズ）
- 現地での撮影・即時登録フロー（将来フェーズ）
- ネイティブフォトライブラリへのアクセス（将来フェーズ）
- ユーザー認証・クラウドストレージ（将来フェーズ）
- 日本以外の地域（将来フェーズ）

## Decisions

### 決定1: `shape-clipper.tsx` は破棄し再設計する

**採用**: 既存コードを流用せず、マップキャンバス中心の設計として新規実装する。

**理由**: `shape-clipper.tsx` は「任意のSVG × 1枚の写真」という汎用ツールとして設計されており、「地理的コレクション」という新しいUXモデルとアーキテクチャが根本的に異なる。クリッピングの概念（SVG clipPath + Canvas書き出し）は再利用するが、コードは再利用しない。

---

### 決定2: 地理データはビルド時に前処理してSVGパス文字列としてバンドルする

**採用**: GeoJSONをビルド時にNode.jsスクリプトで処理し、各エリアのSVGパス文字列と正規化済みbounding boxをJSONとしてバンドルする。

**不採用の代替案**:
- `react-simple-maps` / `d3-geo` をランタイムで使う → 実行時に毎回投影変換が走るが、バンドルサイズと複雑さに対してメリットが少ない
- SVGファイルをそのまま配信する → エリアごとのメタデータ管理が困難

**理由**:
- 離島オフセット（沖縄等）を前処理時に自由に制御できる
- ランタイムコストがほぼゼロ
- エリアのメタデータ（ID・名称・bounding box）を一緒に管理しやすい
- 将来的に市区町村レベルを追加する際も同じパイプラインで対応できる

**データ構造のイメージ**:
```
geo-data/
  japan-prefectures.json   ← 都道府県47件のパスと座標情報
  tokyo-wards.json         ← 東京23区のパスと座標情報
  ...
```
各エントリのスキーマ:
```json
{
  "id": "tokyo",
  "name": "東京都",
  "path": "M 312,145 L 334,152 ...",
  "bbox": { "x": 305, "y": 140, "width": 45, "height": 38 }
}
```

---

### 決定3: クリッピング座標はマップ座標系と独立して正規化する

**採用**: 各エリアのSVGパスをbounding boxで正規化（0〜1のunit空間に変換）し、クリップ編集モーダルでは独自のviewBoxでスケーリングする。

**理由**:
- マップキャンバスは全エリアが800×600等の固定viewBoxに収まる → 各エリアは数十px程度の小さな形状
- クリップ編集モーダルは1エリアを300×300等に拡大表示したい
- 正規化済みパスならどのサイズでも `transform="scale(N)"` で対応できる

---

### 決定4: 状態管理はlocalStorage + IndexedDB を使い分ける

**採用**:
- コレクションのメタデータ・写真の位置/スケール設定 → localStorage
- 写真ファイルのバイナリデータ → IndexedDB（localStorageは5MB制限があるため）

**理由**: 写真ファイルはサイズが大きく、localStorageの容量上限を容易に超える。IndexedDBはバイナリデータの保存に適しており、ブラウザのサポートも広い。

---

### 決定5: エクスポートはCanvas APIで実装する

**採用**: SVGをCanvas要素に描画し、`canvas.toBlob()` でPNGとして書き出す。

**理由**:
- `shape-clipper.tsx` で動作が検証済み
- 解像度を自由に指定できる（長辺1024px以上の要件を満たせる）
- 背景透過（アルファチャンネル）も標準でサポート

### 決定6: コンポーネントライブラリは shadcn/ui を採用する

**採用**: shadcn/ui をベースコンポーネントライブラリとする。必要に応じて daisyUI を補完的に導入する。

**理由**:
- Tailwind CSS とネイティブに統合されており、スタイルの一貫性を保ちやすい
- コンポーネントのコードがプロジェクト内（`/components/ui/`）に配置されるため、Claude が直接読み・編集できる（AI駆動開発との相性が良い）
- Radix UI ベースのため、Dialog・Slider 等の複雑なUIのアクセシビリティが標準で確保されている
- daisyUI は CSS クラスベースであり shadcn/ui と衝突しない。shadcn/ui でカバーしにくいUI（ローディング・バッジ等）を補う形で共存できる

---

### 決定7: 地理データは `/public` に静的配置し、fetch で取得する

**採用**: 前処理済みの地理JSONファイルを `/public/geo-data/` に配置し、クライアントから fetch で取得する。将来は Supabase Storage に移行する。

**理由**:
- JS バンドルにデータを含めるとバンドルサイズが増大する。テンプレートを遅延ロードできる fetch 方式の方が初期表示に有利
- URL を環境変数（`GEO_DATA_BASE_URL`）に抽象化することで、Supabase Storage への移行が URL の変更だけで完結する

```
MVP:    fetch('/geo-data/japan-prefectures.json')
将来:   fetch(`${process.env.GEO_DATA_BASE_URL}/japan-prefectures.json`)
```

## Risks / Trade-offs

| リスク | 緩和策 |
|--------|--------|
| 前処理パイプラインのメンテナンスコスト（データ更新時に再実行が必要） | スクリプトをCIに組み込み、GeoJSONソースの更新を検知したら自動再生成する |
| IndexedDBのデータがブラウザのキャッシュクリアで消える | 将来フェーズでクラウドバックアップを提供。MVPではユーザーに注意表示する |
| 小さい県（香川・大阪等）がクリックしにくい | ズーム機能で対処。最低限のヒットエリア（padding）を設ける |
| Canvas書き出し時のCORS制約（`<image>` タグ経由の写真） | ファイルピッカー経由のローカルファイルはObjectURLを使うためCORS問題は発生しない |

## Open Questions

- GeoJSONデータソース: 国土数値情報（MLIT）か、npmパッケージ（`japan-geojson` 等）か → ライセンスと精度を確認して選定する
- 離島の具体的なオフセット値: 沖縄・伊豆諸島・小笠原のオフセット量は前処理スクリプトのパラメータとして実装時に調整する
- PWA化（Service Worker）: MVPに含めるか将来フェーズにするか
